# Hard Task Discipline

難しいタスクを進めるときの思考の規律。「賢く考える」のではなく「間違いにくい順序で作業する」ことを目的とする。

## 原則

- **理解より先に手を動かさない。ただし理解のためには手を動かす**。コードを読む・小さく実験する・ログを見るのは「着手」ではなく調査であり、早いほどよい。書き換えを始めるのが早すぎるのが問題。
- **自分の結論はデフォルトで疑う**。もっともらしい説明が最初に浮かんだとき、それは「最初に浮かんだ」以上の根拠を持たない。
- **完了の宣言には観測された証拠が要る**。「実装した」と「動くことを確認した」を区別する。

## フェーズ 1: タスクの分解

### 1-1. 完了条件を先に固定する

作業を始める前に「何が観測できたら完了か」を 1〜2 文で書き下す。書けない場合、タスクが曖昧なので調査または質問に切り替える。

- 悪い例: 「ページネーションを改善する」
- 良い例: 「1 万件のデータで該当クエリの EXPLAIN ANALYZE が index scan になり、実行時間が 100ms を切る」

### 1-2. 急所となる不明点 (load-bearing unknown) を特定する

サブタスクを列挙する前に「これが分からないと計画全体が無意味になる不明点」を探す。例:

- 既存コードがそのユースケースを既にサポートしているか
- 外部 API / ライブラリが想定した動作をするか
- データの実際の分布・件数（設計判断が変わる）

急所の不明点は**最初に**潰す。計画を立ててから調べるのではなく、調べてから計画を立てる。

### 1-3. 独立に検証可能な単位に割る

各ステップは「そのステップ単体で成功/失敗が判定できる」粒度にする。判定できないステップは分解が足りないか、順序が間違っている。

- 可逆で安い操作を先に、不可逆で高い操作を後に並べる
- ステップ間の依存を明示する。依存がないものは順序に意味がないと認識しておく（後で並べ替えられる）
- 3 ステップを超える作業は TodoWrite 等で外部化し、頭の中で管理しない

### 1-4. 分解の見直し条件を決めておく

「ステップ 2 でこういう結果が出たら計画ごと見直す」という撤退条件を最初に決める。計画に沈没コストを感じ始めたら危険信号。

## フェーズ 2: 自己検証

### 2-1. 実装直後にやること

1. **書いた変更を自分でもう一度読む**。書いた直後の自分は最も甘い査読者なので、「このコードを他人が PR に出してきたら何を指摘するか」の視点で読む。
2. **変更した挙動そのものを動かす**。テストが通ることと機能が動くことは別。エントリポイントから実際のフローを 1 回通す。
3. **失敗するはずのケースを 1 つ試す**。正常系だけの確認は確認ではない。境界値・空データ・異常系のうち最も壊れやすそうなものを選ぶ。

### 2-2. 調査結果・原因分析の検証

- 結論に至ったら、**独立した第二の証拠**を探す。同じログを二度読むのは独立していない。別の角度（コード側から / データ側から / 履歴側から）で同じ結論が出るか。
- 「自分の仮説が間違っているとしたら、世界はどう見えるはずか」を 1 回考える。その観測をまだしていないなら、する。
- 反証を 1 つも試していない結論は、報告時に「未検証の仮説」と明示する。

### 2-3. 完了報告の規律

- テストが落ちたらその出力ごと報告する。落ちたことを隠して先に進まない
- スキップしたステップは「スキップした」と言う
- 「たぶん動く」を「動く」と報告しない。検証していないなら検証していないと書く

## フェーズ 3: 次の一手の決定

各ステップ完了時に、次の 4 択で分岐する:

1. **予定どおり次のステップへ** — 直前の結果が計画の前提を壊していない場合のみ
2. **計画を修正する** — 新しい情報が前提を変えた場合。修正内容を 1 行で明示してから進む（黙って軌道修正しない）
3. **ユーザーに戻す** — 次の判断が「発見可能な事実」ではなく「ユーザーの意思決定」である場合のみ。事実で埋められる不明点を質問で埋めようとしない
4. **停止して報告する** — 完了条件を満たした、または同じアプローチで 2 回失敗した場合。同じ手を 3 回試さない。2 回失敗したら手法ではなく理解が間違っている可能性を疑い、フェーズ 1 に戻る

### 迷ったときの判定基準

- 「今持っている情報で行動できるか？」→ できるなら行動する。既に確立した事実を再導出しない
- 「この選択は可逆か？」→ 可逆なら試すほうが調べるより速いことが多い
- 「この作業は完了条件に向かっているか？」→ 向かっていない作業（過剰なリファクタ、頼まれていない改善）に気づいたら止める
